## ‚ùì Q17:

## Why is Contravariance not supported in Method Parameters?

### ‚úÖ Deep Answer:

This is a **Type Safety limitation** that C# enforces to prevent Runtime errors.

### ‚ùå If Contravariance was allowed:

<div dir="ltr" style="text-align: left;">

```csharp
class Animal { }
class Dog : Animal { }
class Cat : Animal { }

class Base {
    public virtual void Process(Dog dog) {
        dog.Bark();  // Only Dog can Bark
    }
}

class Derived : Base {
    // Suppose Contravariance was allowed:
    public override void Process(Animal animal) {  // ‚ùå Would error
        // Now animal can be Dog or Cat!
        animal.Bark();  // ‚ùå Cat cannot Bark!
    }
}
```

<div dir="ltr" style="text-align: left;">

### üîç Type Safety Problem:

<div dir="ltr" style="text-align: left;">

```csharp
Base obj = new Derived();
Dog myDog = new Dog();

// Base.Process() expects Dog:
obj.Process(myDog);  

// But Derived.Process() expects Animal!
// What if we send Cat?
Cat myCat = new Cat();
obj.Process(myCat);  // ‚ùå Derived.Process() executes
                      // but Derived.Process() thinks it has Animal
                      // and might call Dog methods ‚Üí Runtime Error!
```

<div dir="ltr" style="text-align: left;">

### ‚úÖ Solution: Interface with Contravariance

<div dir="ltr" style="text-align: left;">

```csharp
// In Interfaces we can have Contravariance:
interface IProcessor<in T> {  // 'in' = Contravariant
    void Process(T item);
}

IProcessor<Animal> processor = new DogProcessor();  // ‚úÖ
// Now we can send Animal:
processor.Process(new Dog());  // ‚úÖ
processor.Process(new Cat());  // ‚úÖ
```

<div dir="ltr" style="text-align: left;">

### üéØ Difference:

- **Virtual Methods**: Contravariance not allowed (Type Safety)
- **Generic Interfaces**: Contravariance with `in` keyword is allowed
- **Delegates**: Contravariance in parameter types is allowed

### üí° Reason:

Virtual Methods must follow **Liskov Substitution Principle**. Contravariance violates this principle.
